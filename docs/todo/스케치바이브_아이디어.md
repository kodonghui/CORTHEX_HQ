# 스케치바이브 (SketchVibe) — 아이디어 기록

> 최초 논의: 2026-02-28
> 대표님 아이디어. 특허 출원 검토 중.

---

## 한 줄 요약

> **손으로 대충 그린 스케치 + 말(자연어)을 동시에 입력하면, AI가 의도를 파악해서 정확한 다이어그램으로 변환하고, 확인 후 실제 구현까지 이어지는 시스템**

---

## 왜 이게 필요한가 (탄생 배경)

자율 AI 에이전트 시스템을 만드는 사람에게 가장 큰 병목은:

> **"머릿속 구조를 AI한테 정확하게 전달하는 것"**

말로만 설명하면 오해가 생기고, 코드로 직접 쓰면 비개발자는 못 한다.
스케치로 그리면서 말하는 게 가장 빠르고 자연스럽다.

실제로 2026-02-28 논의에서 말로 설명하다가 다이어그램 보여주자마자
"어 맞아 그거야!"가 나온 것이 이 아이디어의 직접적 계기.

---

## 핵심 파이프라인

```
① 대표님이 손으로 스케치 (이미지)
        ↓
② AI가 이미지 읽음 → Mermaid 코드로 변환
   (이미지보다 코드가 구조를 더 정확하게 표현)
        ↓
③ Claude가 Mermaid 코드 + 자연어 설명 동시에 읽음
   → 의도 / 방향성 / 맥락 파악
        ↓
④ "아, 이 말씀이시군요~" → 정확한 다이어그램으로 보정
   원본 스케치 위에 덧그리거나 수정해서 보여줌
        ↓
⑤ 대표님 "맞아" 확인 (교정 포인트)
        ↓
⑥ Claude 실제 구현
```

---

## 핵심 기술 통찰 3가지

### 1. Mermaid를 중간 언어로 쓰는 것
이미지를 바로 읽는 게 아니라 **Mermaid 코드로 변환한 뒤 읽는다.**

왜 이게 중요한가:
- 이미지 직접 읽기: 정확도 80% (박스 겹침, 글씨 흐릿하면 실수)
- Mermaid 코드: 정확도 100% (구조가 명확하게 텍스트로 표현됨)

### 2. 자연어 + 스케치 동시 입력 (멀티모달)
자연어만: AI가 형태를 모름
스케치만: AI가 의도/맥락을 모를 수 있음
**둘 동시에**: 시각 정보 + 의미 정보가 합쳐져 의도 추론 정확도가 비약적으로 올라감

### 3. 중간 확인 단계 (⑤번)
이미지 → AI → 바로 구현: 잘못 이해해도 대표님이 모름
이미지 → Mermaid → "이렇게 이해했습니다" → **대표님 확인** → 구현

> 비행기 자동항법이 아니라, 기장이 경로를 확인하는 체크포인트가 있는 구조.
> 이게 단순 편의기능이 아니라 신뢰성의 핵심.

## 다이어그램 종류 (이 시스템이 처리하는 것)

> **핵심 UX 결정 (2026-02-28)**: 사용자는 "연결도인지 흐름도인지" 구분할 필요 없음.
> 그냥 그리면 → AI가 자동 판단 → 맞는 형태로 출력.
> 타입 선택 UI 없음. "그려서 올리면 끝"이 전부.

내부적으로 처리하는 두 종류:

| 종류 | 설명 | 예시 |
|------|------|------|
| **연결도** | 파일/모듈/시스템이 서로 어떻게 연결됐는지 관계도 | arm_server.py → trading_engine.py → kis_client.py |
| **흐름도** | 버튼 클릭 → 어느 파일 → 결과까지 순서도 | 사용자 클릭 → handler → engine → DB → 화면 |

사용자는 타입을 모르고 그려도 된다. AI가 스케치 + 자연어로 자동 판단한다.

### MVP Phase 구성 (2026-02-28 확정)

| Phase | 내용 | 사용자 경험 |
|-------|------|------------|
| **Phase 1** | 어떤 그림이든 받아서 처리 (연결도 + 흐름도 모두) | "그려서 올리면 → 결과 나옴" |
| **Phase 2** | 더 복잡한 그림 처리, 정확도 향상 | Phase 1과 동일한 UX, 내부 정확도만 향상 |

Phase 1 / Phase 2 구분은 내부 구현 수준의 차이. 사용자 화면은 동일.

> 특허는 가능, 이름(상표)은 재검토 필요.

## 입력 인터페이스 — 웹 캔버스 + Claude MCP (2026-02-28 확정)

### 확정된 방향
- **입력 방식**: 웹 캔버스에서 직접 그림 (사진 업로드 아님)
- **전달 방식**: **Claude MCP** 통해 실시간 공유

### Claude MCP란?
MCP(Model Context Protocol) = Anthropic이 만든 "외부 도구를 Claude에 꽂는 표준 규격".
Claude Code가 캔버스 상태를 직접 구독(읽기)할 수 있게 해주는 연결 방식.

파일 저장/복붙/업로드 없이 → Claude가 캔버스를 직접 봄.

### MCP 파이프라인 (업데이트된 전체 흐름)

```
① 대표님이 CORTHEX 웹 캔버스에 스케치
        ↓
② 캔버스 상태가 MCP 서버를 통해 Claude Code에 실시간 노출
        ↓
③ Claude Code가 캔버스 읽음 + 자연어 설명 동시 수신
        ↓
④ AI가 다이어그램 타입 자동 판단 (연결도/흐름도 — 사용자는 모름)
        ↓
⑤ Mermaid 코드로 변환 (정확도 100% 확보)
        ↓
⑥ "이렇게 이해했습니다" → 정확한 다이어그램 보정
        ↓
⑦ 대표님 "맞아" 확인
        ↓
⑧ Claude 실제 구현
```

### 왜 Claude MCP인가 (경쟁 우위)

| | Claude MCP | 다른 AI 툴 |
|---|---|---|
| MCP 표준 | Anthropic이 직접 만든 규격 | 나중에 따라옴 |
| 호환 도구 수 | 수백 개 (Notion, GitHub, Slack 등) | 초기 단계 |
| 생태계 | 계속 확장 중 | 미성숙 |
| 보안 검증 | 미 국방부 프로젝트 공급 계약 (2025~) | 검증 없음 |

> 스케치바이브를 Claude MCP 전용으로 설계하면:
> 다른 AI가 MCP 생태계 따라오려면 1~2년 걸림 → 그 기간이 독점 우위 구간.

### 기존 자산 활용
- Nexus Mode 2 Drawflow 캔버스 → 스케치 입력 인터페이스로 재활용 가능
- CORTHEX 기존 MCP 서버 구조 (Notion MCP 등) → 캔버스용 MCP 서버 추가하는 패턴 동일
