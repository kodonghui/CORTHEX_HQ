# Product Brief: CORTHEX_HQ v2

## Executive Summary

CORTHEX v2는 현재 1인용 AI 에이전트 지휘 시스템(v1)을 **멀티유저 SaaS 플랫폼**으로 재구축하는 프로젝트다. CEO뿐 아니라 인간 직원(H)이 로그인하여 자기 부하 AI 에이전트를 지휘하고, 상위자에게 보고할 수 있는 구조를 만든다. 나아가 다른 회사에도 이 플랫폼을 제공할 수 있는 멀티 컴퍼니 구조를 갖춘다.

---

## Core Vision

### Problem Statement

CORTHEX v1은 대표(CEO) 1인 전용으로 설계되었다. 새로운 인간 직원(H)을 추가할 때마다 코드를 복사-붙여넣기하여 별도 환경을 구성해야 하며, 이 과정에서 반복적인 버그가 발생한다:

- H 화면에 "대표님"이라는 하드코딩된 인사말이 뜸
- CEO의 KIS 투자 연동 정보가 H에게 노출됨 (데이터 격리 실패)
- H의 Claude CLI 연동이 CEO의 것과 뒤섞임
- 버그 수정에 매번 과도한 시간 소요

### Problem Impact

- 인간 직원 1명 추가할 때마다 수일간의 버그 수정 작업 필요
- 조직 확장이 기술적 병목에 막혀 있음
- v1 코드베이스의 구조적 한계로 근본 해결 불가능

### Why Existing Solutions Fall Short

- **v1 패치 접근**: role if/else, showSections 등으로 사용자별 분기를 시도했으나 버그만 증가. 구조적으로 멀티유저를 고려하지 않고 만든 코드에 덧붙이기는 한계
- **범용 SaaS 도구** (Slack, Notion 등): AI 에이전트 지휘 + 조직 계층 구조 + CLI 연동이라는 CORTHEX 고유의 요구사항을 충족 불가
- **AI 에이전트 플랫폼** (CrewAI, AutoGen 등): 멀티유저/멀티테넌트 지원 없음. 조직도 기반 위임-보고 구조 없음

### Proposed Solution

서버에 CORTHEX v2를 새로 구축한다. 핵심 설계 원칙:

1. **멀티 컴퍼니 SaaS** — 관리자 모드에서 여러 회사(CORTHEX-HQ, company_example1, ...) 관리
2. **인간(H) + AI 에이전트 조직** — H가 ID/PW로 로그인, 자기 부하 AI에게 지휘, 상위자에게 보고
3. **CLI 단위 격리** — 각 H의 AI 에이전트는 H의 Claude CLI 계정 + H가 연동한 API key로만 작동. 비용·협업·데이터 전부 CLI 경계로 격리
4. **비서실장 오케스트레이션** — CEO는 비서실장이 정리한 최종 보고서만 받음. 위임↔보고 구조
5. **본부별 정보 DB** — 에이전트는 자기 본부 정보만 참조
6. **3단계 도구 구조** — 플랫폼 공통 도구 → 회사별 도구 → 본부별 도구 할당. 관리자가 ON/OFF
7. **관리자 모드** — 부서/에이전트/도구를 실시간 추가·수정·삭제 (스케치바이브 등 활용)
8. **웹 + 모바일** — 별도 모바일 페이지(m.도메인)

### Key Differentiators

- **"사람 1명 = CLI 1개 = 에이전트 N명 = 독립 비용/데이터"** — 다른 AI 플랫폼에 없는 인간-AI 혼합 조직 모델
- **같은 본부에 다른 CLI 소속 에이전트 공존 가능**, 단 협업은 CLI 기준으로 격리 — 파견 직원 모델
- **비개발자(CEO)가 관리자 모드에서 조직 구조를 직접 수정** — 코드 변경 없이 부서/에이전트/도구 추가
- **v1에서 검증된 핵심 기능**(에이전트 소울, 위임-보고, 도구 자동호출)을 구조적으로 재설계하여 이식

## Target Users

### Primary Users

**페르소나 1: 동희 (회사 유저 — CEO)**
- **역할**: CORTHEX-HQ의 대표. 유저 앱에서는 다른 H와 동일한 일반 유저
- **동기**: AI 에이전트 조직을 통해 1인 기업의 한계를 돌파. 인간 직원을 채용해 조직 확장
- **핵심 행동**:
  - 비서실장에게 업무 지시 → 최종 보고서만 받음
  - SNS 게시 승인
  - AI 에이전트와 멀티턴 채팅으로 업무 수행
- **성공 순간**: "H를 채용해서 본부장으로 앉혔는데, 코드 한 줄 안 건드리고 바로 됐다"

**페르소나 2: 민지 (인간 직원 H — 본부장급)**
- **역할**: CEO에게 채용되어 특정 본부를 맡은 인간 직원
- **동기**: AI 부하직원들을 지휘해서 업무 성과를 내고, CEO에게 보고
- **하루 흐름**:
  1. 출근 → ID/PW 로그인
  2. 전날 퇴근 전 AI에게 시켜놓은 업무 보고서 확인
  3. SNS 게시물 승인 처리
  4. 오늘 해야 할 일 확인
  5. AI 에이전트들과 채팅으로 업무 수행 (멀티턴 + 세션 기능)
  6. 퇴근 전 AI에게 야간 업무 지시
- **핵심 니즈**: 자기 CLI로 연동된 자기 에이전트만 보이고, 자기 본부 데이터만 접근
- **성공 순간**: "내 부하 AI 3명이 밤새 분석해놓은 보고서를 아침에 확인하고 바로 CEO에게 보고했다"

### Secondary Users

**관리자 (별도 콘솔, 별도 계정)**
- **역할**: 플랫폼 전체 운영자. 유저 앱과 완전 분리된 관리자 콘솔 사용
- **핵심 행동**:
  - 회사 CRUD (생성/수정/삭제)
  - 직원 계정 발급 (ID/PW)
  - CLI / API key 연동
  - 부서/에이전트/도구 배정 및 ON/OFF
  - 향후: 외부 회사에 SaaS 서비스 시 별도 관리자가 될 수 있음
- **핵심 원칙**: CEO도 유저 앱에서는 "한 명의 유저". 관리 업무는 별도 콘솔에서만.

### User Journey

**관리자 여정**: 관리자 콘솔 로그인 → 회사 생성 → 직원 계정 발급 → CLI/API key 연동 → 부서 생성 → 에이전트 배정 → 도구 할당 → 완료 (H가 로그인하면 바로 업무 시작 가능)

**CEO/H 여정**: 유저 앱 로그인 → "안녕하세요 {이름}님" → 부하 AI 확인 → 채팅으로 업무 지시 → 보고서 수신 → (CEO의 경우) 비서실장 최종 보고서 확인

### Party Mode 검토 결과 (BMAD 팀 합의)

| 항목 | 결정 |
|------|------|
| 관리자 콘솔 | 유저 앱과 **완전 분리** (별도 URL, 별도 계정) |
| CEO 지위 | 유저 앱에서는 **일반 유저와 동일** |
| H 온보딩 | 관리자가 **선세팅** (CLI, 에이전트, 도구) → H는 로그인만 하면 바로 시작 |
| UI | **1벌**. 모든 유저 동일 화면, 데이터/CLI/권한만 다름 |
| 첫 화면 | "안녕하세요 {이름}님" + 부하 AI 준비됨 + 문서함 비어있음 |

## Success Metrics

### 유저 성공 지표

| 지표 | 측정 기준 |
|------|-----------|
| **블로커 버그 0건** | H가 업무를 완료하는 데 방해가 되는 버그 0건 (사소한 UI 이슈 제외) |
| **독립 워크스페이스** | H의 화면에 다른 유저의 데이터가 단 한 건도 노출되지 않음 |
| **CLI 격리** | H의 에이전트가 오직 H의 CLI로만 작동. 비용/데이터 혼선 0건 |
| **세션 연속성** | H가 퇴근 후 다음 날 와서 어제 대화를 이어갈 수 있음 (클로드 데스크탑 앱 수준) |
| **H 설정 최소화** | 관리자가 CLI/공용API 전부 선세팅. H는 개인 API key(KIS 등)만 직접 등록 |

### API Key 등록 경계

| 등록 주체 | 대상 |
|-----------|------|
| **관리자** | CLI 토큰, 회사 공용 도구 API (노션, SNS 등) |
| **H 본인** | 개인 API key (KIS 증권 등 개인 계정) |

### Business Objectives

| 목표 | 기준 |
|------|------|
| **직원 추가 소요 시간** | 관리자 콘솔에서 5분 이내에 새 H 세팅 완료 (v1: 수일) |
| **데이터 격리 위반** | 0건. 자동화 격리 테스트 배포마다 실행 |
| **v1 기능 이식률** | v1 핵심 기능(채팅, 위임-보고, 도구호출, SNS, 전략실) 100% 이식 |
| **외부 판매 준비** | 멀티 컴퍼니 구조로 외부 회사 온보딩 가능한 상태 |

### Key Performance Indicators

| KPI | 비고 |
|-----|------|
| **H 첫 업무 완료 시간** | 계정 발급 → 로그인 → AI에게 첫 지시 → 결과 수신까지 소요 시간 |
| **블로커 버그 건수** | 배포 후 첫 2주간 업무 방해 버그 수 (목표: 0건) |
| **관리자 조직 수정 시간** | 부서/에이전트/도구 추가에 걸리는 시간 (목표: 5분 이내) |
| **테넌트 격리 테스트 통과율** | 자동화 테스트 배포마다 100% 통과 |
| 3개월/12개월 비즈니스 목표 | 추후 결정 |

## MVP Scope

### 스코프 원칙

**"전부 넣되, 이식 순서는 나눈다."** v1에서 이미 검증된 기능을 빼고 시작하면 의미 없음. Phase는 "뭘 빼자"가 아니라 "뭘 먼저 검증하자".

### P1: 뼈대 (최우선 — 이게 없으면 아무것도 안 됨)

| 기능 | 설명 |
|------|------|
| **관리자 콘솔** | 별도 URL. 회사/직원/CLI/부서/에이전트 CRUD |
| **유저 앱 로그인** | ID/PW 인증 + "안녕하세요 {이름}님" + 독립 워크스페이스 |
| **CLI 격리** | 에이전트 실행 = 해당 유저 CLI 토큰으로만 작동 |
| **AI 에이전트 채팅** | 멀티턴 + 세션 연속성 (클로드 데스크탑 앱 수준) |
| **비서실장 오케스트레이션** | 위임→보고 구조. CEO는 최종 보고서만 수신 |
| **테넌트 격리 테스트** | 자동화 테스트 배포마다 실행 |

**P1 완료 시점**: 대표님이 v2에서 "첫 채팅" 가능. H 로그인해서 AI에게 첫 지시 가능.

### P2: 업무 (실제 업무 수행에 필요)

| 기능 | 설명 |
|------|------|
| **전략실** | KIS 매매 연동 등. H가 개인 API key 직접 등록 |
| **도구 호출** | 에이전트가 도구 자동호출. 3단계 할당 (공통→회사→본부) |
| **본부별 정보 DB** | 에이전트는 자기 본부 정보만 참조 |
| **보고서/문서함** | 에이전트 보고서 저장 + 열람. 유저별 격리 |

**P2 완료 시점**: v1 대체 가능. 실제 업무 수행 가능.

### P3: 운영 (v1 나머지 기능 이식)

| 기능 | 설명 |
|------|------|
| **SNS 발행/승인** | 콘텐츠 생성 → 승인 → 자동 발행 |
| **작전일지** | 활동 로그 기록 + 열람 |
| **대시보드** | 비용 모니터링 + 에이전트 상태 |
| **텔레그램 연동** | 모바일 지휘 + 알림 |
| **에이전트 소울 편집** | 에이전트 성격/전문분야 설정 |

**P3 완료 시점**: v1 완전 이전. v1 종료 가능.

### Out of Scope (P4 — 나중에)

| 기능 | 이유 |
|------|------|
| **모바일 전용 (m.도메인)** | 웹 반응형으로 우선 대체 |
| **멀티 컴퍼니 외부 판매** | 먼저 CORTHEX-HQ만 안정화 |
| **스케치바이브 조직 편집** | 관리자 콘솔 UI로 우선 대체 |
| **NEXUS 3D 시각화** | P1~P3 안정화 후 |

### MVP Success Criteria

- P1 완료 → 대표님 첫 테스트 통과 (H 로그인 + 채팅 + 격리 확인)
- P2 완료 → v1 기능 대체 가능 확인
- P3 완료 → v1 완전 종료 결정

### Party Mode 검토 결과 (BMAD 팀 합의 — 스코프)

| 논점 | 결정 |
|------|------|
| 전부 넣기 vs MVP | **전부 넣되 이식 순서(P1→P3) 유지** |
| Phase의 의미 | "빼는 것"이 아니라 **"검증 순서"** |
| P4 | 나중에. P1~P3 안정화 후 착수 |
