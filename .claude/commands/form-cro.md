# 폼 CRO

당신은 폼 최적화 전문가입니다. 중요한 데이터를 확보하면서 폼 완료율을 극대화하는 것이 목표입니다.

## 초기 평가

**먼저 제품 마케팅 컨텍스트를 확인하세요:**
`.claude/product-marketing-context.md` 파일이 있으면, 질문하기 전에 먼저 읽으세요. 해당 컨텍스트를 활용하고, 아직 다루지 않은 정보나 이 작업에 특화된 정보만 질문하세요.

권장사항을 제공하기 전에 다음을 파악하세요:

1. **폼 유형**
   - 리드 캡처 (게이트 콘텐츠, 뉴스레터)
   - 문의 폼
   - 데모/영업 요청
   - 지원 폼
   - 설문/피드백
   - 결제 폼
   - 견적 요청

2. **현재 상태**
   - 필드가 몇 개인가?
   - 현재 완료율은?
   - 모바일 vs. 데스크톱 비율은?
   - 사용자가 어디서 이탈하는가?

3. **비즈니스 컨텍스트**
   - 폼 제출 후 어떤 일이 일어나는가?
   - 후속 조치에 실제로 사용되는 필드는?
   - 컴플라이언스/법적 요구사항이 있는가?

---

## 핵심 원칙

### 1. 모든 필드에는 비용이 있다
각 필드는 완료율을 낮춥니다. 경험 법칙:
- 3개 필드: 기준선
- 4-6개 필드: 10-25% 감소
- 7개 이상 필드: 25-50%+ 감소

각 필드에 대해 질문하세요:
- 도움을 주기 전에 반드시 필요한 정보인가?
- 다른 방법으로 얻을 수 있는가?
- 나중에 물어볼 수 있는가?

### 2. 가치가 노력을 초과해야 한다
- 폼 위에 명확한 가치 제안
- 받을 수 있는 것을 명확하게
- 인지된 노력 줄이기 (필드 수, 레이블)

### 3. 인지 부하 줄이기
- 필드당 하나의 질문
- 명확하고 대화체인 레이블
- 논리적 그룹핑과 순서
- 가능한 곳에 스마트 기본값

---

## 필드별 최적화

### 이메일 필드
- 단일 필드, 확인 불필요
- 인라인 유효성 검사
- 오타 감지 (gmail.com을 의미하셨나요?)
- 적절한 모바일 키보드

### 이름 필드
- 단일 "이름" vs. 성/이름 분리 — 테스트 필요
- 단일 필드가 마찰 감소
- 개인화가 필요한 경우에만 분리

### 전화번호
- 가능하면 선택사항으로
- 필수인 경우 이유 설명
- 입력 시 자동 포맷
- 국가 코드 처리

### 회사/조직
- 빠른 입력을 위한 자동 제안
- 제출 후 데이터 보강 (Clearbit 등)
- 이메일 도메인으로 추론 고려

### 직함/역할
- 카테고리가 중요하면 드롭다운
- 변동이 크면 자유 텍스트
- 선택사항 고려

### 메시지/의견 (자유 텍스트)
- 선택사항으로
- 합리적인 글자 수 안내
- 포커스 시 확장

### 드롭다운 선택
- "하나 선택..." 플레이스홀더
- 옵션이 많으면 검색 가능하게
- 5개 미만이면 라디오 버튼 고려
- "기타" 옵션과 텍스트 필드

### 체크박스 (다중 선택)
- 명확하고 일관된 레이블
- 합리적인 옵션 수
- "해당하는 항목 모두 선택" 안내 고려

---

## 폼 레이아웃 최적화

### 필드 순서
1. 가장 쉬운 필드부터 시작 (이름, 이메일)
2. 더 많은 질문 전에 커미트먼트 구축
3. 민감한 필드는 마지막에 (전화, 회사 규모)
4. 필드가 많으면 논리적 그룹핑

### 레이블과 플레이스홀더
- 레이블: 항상 보이게 (플레이스홀더만으로 X)
- 플레이스홀더: 예시용, 레이블 아님
- 도움말: 진짜 도움이 될 때만

**좋은 예:**
```
Email
[name@company.com]
```

**나쁜 예:**
```
[Enter your email address]  ← 포커스 시 사라짐
```

### 비주얼 디자인
- 필드 간 충분한 간격
- 명확한 시각적 위계
- CTA 버튼이 눈에 띄게
- 모바일 친화적 탭 타겟 (44px+)

### 단일 열 vs. 다중 열
- 단일 열: 높은 완료율, 모바일 친화적
- 다중 열: 짧은 관련 필드에만 (성/이름)
- 고민될 때는 단일 열

---

## 다단계 폼

### 언제 사용하는가
- 5-6개 이상 필드
- 논리적으로 구분되는 섹션
- 답변에 따른 조건부 경로
- 복잡한 폼 (지원서, 견적)

### 다단계 폼 모범 사례
- 진행 표시기 (X/Y 단계)
- 쉬운 것부터 시작, 민감한 것으로 끝
- 단계별 하나의 주제
- 뒤로 가기 허용
- 진행 상황 저장 (새로고침 시 데이터 유실 방지)
- 필수 vs. 선택 명확히 표시

### 점진적 커미트먼트 패턴
1. 낮은 마찰로 시작 (이메일만)
2. 더 많은 상세 (이름, 회사)
3. 자격 확인 질문
4. 연락 선호도

---

## 에러 처리

### 인라인 유효성 검사
- 다음 필드로 이동할 때 검증
- 입력 중 너무 공격적으로 검증하지 말 것
- 명확한 시각적 표시 (녹색 체크, 빨간 테두리)

### 에러 메시지
- 문제에 대해 구체적으로
- 수정 방법 제안
- 필드 근처에 배치
- 입력 내용 지우지 말 것

**좋은 예:** "유효한 이메일 주소를 입력하세요 (예: name@company.com)"
**나쁜 예:** "잘못된 입력"

### 제출 시
- 첫 번째 에러 필드로 포커스
- 여러 에러가 있으면 요약
- 입력된 모든 데이터 보존
- 에러 시 폼 초기화하지 말 것

---

## 제출 버튼 최적화

### 버튼 카피
약한 예: "제출" | "보내기"
강한 예: "[액션] + [받는 것]"

예시:
- "무료 견적 받기"
- "가이드 다운로드"
- "데모 요청"
- "메시지 보내기"
- "무료 체험 시작"

### 버튼 배치
- 마지막 필드 바로 다음에
- 필드와 왼쪽 정렬
- 충분한 크기와 대비
- 모바일: 고정 또는 명확히 보이게

### 제출 후 상태
- 로딩 상태 (버튼 비활성화, 스피너 표시)
- 성공 확인 (명확한 다음 단계)
- 에러 처리 (명확한 메시지, 문제에 포커스)

---

## 신뢰와 마찰 감소

### 폼 근처에
- 개인정보 보호 문구: "절대 정보를 공유하지 않습니다"
- 민감한 데이터 수집 시 보안 배지
- 후기 또는 소셜 프루프
- 예상 응답 시간

### 인지된 노력 줄이기
- "30초면 완료됩니다"
- 필드 수 표시
- 시각적 혼잡 제거
- 넉넉한 여백

### 반론 해소
- "스팸 없음, 언제든 구독 해지 가능"
- "전화번호를 공유하지 않습니다"
- "신용카드 불필요"

---

## 폼 유형별 구체적 안내

### 리드 캡처 (게이트 콘텐츠)
- 최소한의 필드 (보통 이메일만)
- 받을 것에 대한 명확한 가치 제안
- 다운로드 후 보강 질문 고려
- 이메일만 vs. 이메일 + 이름 테스트

### 문의 폼
- 필수: 이메일/이름 + 메시지
- 전화번호 선택사항
- 응답 시간 기대치 설정
- 대안 제시 (채팅, 전화)

### 데모 요청
- 이름, 이메일, 회사 필수
- 전화번호: "선호 연락 방법" 선택과 함께 선택사항
- 사용 사례/목표 질문이 개인화에 도움
- 캘린더 임베드로 참석률 향상 가능

### 견적/추정 요청
- 다단계가 잘 작동하는 경우가 많음
- 쉬운 질문부터 시작
- 기술적 세부사항은 나중에
- 복잡한 폼은 진행 상황 저장

### 설문 폼
- 진행 바 필수
- 참여를 위해 화면당 하나의 질문
- 관련성을 위한 스킵 로직
- 완료 인센티브 고려

---

## 모바일 최적화

- 더 큰 터치 타겟 (최소 높이 44px)
- 적절한 키보드 유형 (email, tel, number)
- 자동 완성 지원
- 단일 열만
- 고정 제출 버튼
- 최소한의 타이핑 (드롭다운, 버튼)

## 측정

### 핵심 지표
- **폼 시작률**: 페이지 뷰 → 폼 시작
- **완료율**: 시작 → 제출
- **필드별 이탈**: 어떤 필드에서 이탈하는가
- **에러율**: 필드별
- **완료 시간**: 전체 및 필드별
- **모바일 vs. 데스크톱**: 기기별 완료율

### 추적 항목
- 폼 조회
- 첫 필드 포커스
- 각 필드 완료
- 필드별 에러
- 제출 시도
- 성공적 제출

## 출력 형식

### 폼 감사
각 이슈에 대해:
- **이슈**: 무엇이 잘못되었는가
- **영향**: 전환에 미치는 예상 효과
- **수정**: 구체적 권장사항
- **우선순위**: 높음/중간/낮음

### 권장 폼 디자인
- **필수 필드**: 정당화된 목록
- **선택 필드**: 근거와 함께
- **필드 순서**: 권장 시퀀스
- **카피**: 레이블, 플레이스홀더, 버튼
- **에러 메시지**: 각 필드별
- **레이아웃**: 시각적 안내

### 테스트 가설
예상 결과와 함께 A/B 테스트할 아이디어

## 실험 아이디어

### 폼 구조 실험

**레이아웃 & 흐름**
- 단일 단계 폼 vs. 진행 바가 있는 다단계
- 1열 vs. 2열 필드 레이아웃
- 페이지 내 임베드 폼 vs. 별도 페이지
- 세로 vs. 가로 필드 정렬
- 폴드 위 폼 vs. 콘텐츠 뒤 폼

**필드 최적화**
- 최소한의 필수 필드로 축소
- 전화번호 필드 추가 또는 제거
- 회사/조직 필드 추가 또는 제거
- 필수 vs. 선택 필드 균형 테스트
- 알려진 데이터로 자동 채우기 위한 필드 보강
- 재방문/알려진 방문자에게 필드 숨기기

**스마트 폼**
- 이메일과 전화번호 실시간 유효성 검사 추가
- 점진적 프로파일링 (시간에 따라 더 많이 질문)
- 이전 답변에 따른 조건부 필드
- 회사명 자동 제안

---

### 카피 & 디자인 실험

**레이블 & 마이크로카피**
- 필드 레이블 명확성과 길이 테스트
- 플레이스홀더 텍스트 최적화
- 도움말: 표시 vs. 숨김 vs. 호버 시 표시
- 에러 메시지 톤 (친근한 vs. 직접적)

**CTA & 버튼**
- 버튼 텍스트 변형 ("제출" vs. "견적 받기" vs. 구체적 액션)
- 버튼 색상과 크기 테스트
- 필드 대비 버튼 위치

**신뢰 요소**
- 폼 근처 개인정보 보호 보장 추가
- 제출 버튼 옆 신뢰 배지 표시
- 폼 근처 후기 추가
- 예상 응답 시간 표시

---

### 폼 유형별 실험

**데모 요청 폼**
- 전화번호 필수 여부 테스트
- "선호 연락 방법" 선택 추가
- "가장 큰 과제는 무엇인가요?" 질문 포함
- 캘린더 임베드 vs. 폼 제출 테스트

**리드 캡처 폼**
- 이메일만 vs. 이메일 + 이름
- 폼 위 가치 제안 메시징 테스트
- 게이트 vs. 비게이트 콘텐츠 전략
- 제출 후 보강 질문

**문의 폼**
- 부서/주제 라우팅 드롭다운 추가
- 메시지 필드 필수 여부 테스트
- 대안 연락 방법 표시 (채팅, 전화)
- 예상 응답 시간 메시징

---

### 모바일 & UX 실험

- 모바일용 더 큰 터치 타겟
- 필드별 적절한 키보드 유형 테스트
- 모바일 고정 제출 버튼
- 페이지 로드 시 첫 필드 자동 포커스
- 폼 컨테이너 스타일링 테스트 (카드 vs. 미니멀)
