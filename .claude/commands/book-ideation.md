# 북 아이디에이션

초기 아이디어를 안내된 다중 세션 개발을 통해 구조화된 논픽션 북 콘셉트로 변환합니다.

## 핵심 철학

**모든 결정은 독자를 위한 것.** 질문은 결코 "내가 무엇을 말하고 싶은가?"가 아니라 "독자에게 어떤 변환이 필요하며, 이 책이 어떻게 전달할 수 있는가?"입니다.

이 스킬은 일반적인 브레인스토밍과 책 아키텍처 사이에 위치합니다. 역할은 원재료를 검증, 시장 테스트, 아키텍처 설계가 가능한 구조화된 콘셉트로 정제하는 것입니다.

**목표는 책 아웃라인이 아닙니다.** 목표는 책이 존재해야 하는지와 무엇을 달성해야 하는지를 결정하는 8가지 근본 요소에 대한 명확성입니다.

## 입력 유형

이 스킬이 받는 것:

- 초기 아이디어 (한 문장 또는 한 문단)
- `brainstorm` 스킬의 브레인스톰 문서
- 제텔카스텐 노트 또는 리서치 모음
- 정제가 필요한 기존 부분적 콘셉트
- 탐색할 가치가 있는 "샤워 생각"

## 세션 흐름

### 세션 시작

매 세션 시작 시:

1. **새로운 것인가 계속인가?**
   - 계속인 경우: 최신 북 콘셉트 문서 버전 요청
   - 새로운 경우: 초기화로 진행

2. **무엇이 있는가?**
   - 사용자에게 원재료 공유 요청 (브레인스톰 문서, 노트, 아이디어)
   - 읽고 이미 명확한 것 vs. 미발달된 것 식별

3. **세션 목표**
   - 오늘 어떤 요소를 개발해야 하는가?
   - 깊은 탐색인가 빠른 진전인가?

### 8가지 요소

대화를 통해 개발하되, 심문하지 않습니다. 데이터를 추출하지 말고 통찰을 표면화합니다.

#### 1. 독자

_구체적으로 누구를 위한 것인가?_

인구통계학을 넘어섭니다. 이해할 것:

- 그들의 현재 상황 (삶/일에서 무슨 일이 일어나고 있는지)
- 도움이 되지 않는 현재 믿음
- 이미 시도했지만 효과가 없었던 것
- 스스로 해결하지 못하게 막는 것
- 이 책을 집어들게 하는 것 (트리거 순간)

**탐색:** "이상적인 독자가 서점에 들어갔다면, 어떤 코너에 있을까요? 무엇을 느끼고 있을까요? 아마존에서 방금 어떤 검색을 했을까요?"

#### 2. 변환

_읽고 난 후 어디에 있을 것인가?_

독자는 포인트 A에서 시작하여 포인트 B에서 끝납니다. 둘 다 명확히 정의:

- 무엇을 다르게 믿을 것인가?
- 무엇을 할 수 있게 될 것인가?
- 어떻게 느낄 것인가?
- 무엇을 그만둘 것인가?

**A와 B 사이의 격차가 책의 존재 이유.**

#### 3. 핵심 논제

_하나의 큰 아이디어는?_

이것은 주제가 아니라 주장입니다. 논제는 입장을 취합니다. 테스트:

- 누군가 이것에 동의하지 않을 수 있는가? (불가능하면 너무 명백)
- 통념에 도전하는가? (반통념적 논제가 더 설득력 있음)
- 한 문장으로 진술할 수 있는가?

**템플릿:** "대부분의 사람들은 [통념]이라고 믿지만, 실제로는 [당신의 논제]인데, [핵심 통찰] 때문이다."

#### 4. 저자 각도

_왜 당신이 이것을 쓰는가?_

신뢰성의 원천:

- 경험 (이것을 살아왔음)
- 전문성 (이것을 연구/실천해왔음)
- 접근 (다른 사람에게 없는 정보가 있음)
- 관점 (다른 사람이 놓치는 것을 봄)

**네 가지 모두 필요하지 않습니다. 적어도 하나가 설득력 있으면 됩니다.**

#### 5. 관건

_왜 이것이 중요한가? 왜 지금인가?_

명확히 할 것:

- 이 책을 읽지 않는 비용은?
- 이 메시지에 왜 지금이 적기인가?
- 현재 상태를 유지하면 독자에게 무엇이 위험한가?

관건이 긴급성을 만듭니다. 관건 없이 책은 "있으면 좋은 것"에 불과합니다.

#### 6. 핵심 개념

_논제를 뒷받침하는 3-7개의 주요 아이디어는?_

이것들은 구성 요소 — 논제를 신뢰성 있고 실행 가능하게 만드는 프레임워크, 원칙, 이야기, 또는 통찰. 챕터나 섹션이 될 것입니다.

20개 아이디어를 나열하지 마세요. 우선순위를 강제하세요. 필수 vs. 있으면 좋은 것?

#### 7. 적

_이 책은 무엇에 반대하는가?_

모든 훌륭한 논픽션 책에는 빌런이 있습니다:

- 마인드셋 ("허슬 문화")
- 관행 ("인박스 제로")
- 통념 ("더 많은 정보 = 더 나은 결정")
- 경쟁 접근법 ("디지털 노트 테이킹")

**적은 대조를 통해 논제를 명확히 합니다.**

#### 8. 약속

_한 문장으로, 독자가 무엇을 얻는가?_

이것이 책의 가치 제안입니다:

- 구체적이어야 함 ("삶을 개선한다"가 아닌)
- 믿을 수 있어야 함 ("30일 안에 백만장자가 된다"가 아닌)
- 변환을 방법에 연결

**템플릿:** "이 책은 [독자]가 [방법/통찰]을 통해 [변환]을 달성하도록 도울 것입니다."

### 세션 진행 중

**협업 행동:**

- 암시적인 것을 표면화: "실제로 말하고 싶은 것은..."
- 약한 요소에 도전: "이 논제가 충분히 반통념적이지 않다고 생각합니다. 이유는..."
- 요소를 연결: "당신의 적은 독자가 ... 일 수 있음을 시사합니다"
- 구체성을 압박: "이 독자의 예를 들어줄 수 있나요?"

**요소에 막혔을 때:**

- 건너뛰고 나중에 돌아오기 (다른 요소가 명확히 해줄 수 있음)
- "이상적 독자 인터뷰" 기법 사용 (독자를 인터뷰한다고 상상)
- "안티 북" 기법 시도 (반대 책은? 누구를 위한 것?)

### 세션 종료

항상 다음으로 마무리:

1. **요소 상태** — 어떤 요소가 확고한지, 개발 중인지, 아직 초기인지?
2. **핵심 발견** — 이번 세션에서 무엇을 발견했나?
3. **숙성 질문** — 사용자가 곰곰이 생각해볼 것은?
4. **버전 생성** — 다음 버전의 북 콘셉트 문서 생성

## 빠른 캡처 모드

시간이 부족할 때 빠른 콘셉트 캡처:

1. 사용자가 초기 아이디어 공유
2. 추출: 독자 (대략), 변환 (대략), 논제 (대략), 약속 (한 문장)
3. 최소 v1 문서 생성
4. 주석: "빠른 캡처 — 향후 세션에서 확장"

## 준비 기준

북 콘셉트 문서가 검증 준비된 시점:

| 요소         | 준비 테스트                                             |
| ------------ | ------------------------------------------------------- |
| 독자         | 카테고리가 아닌 구체적 인물로 묘사 가능                 |
| 변환         | 감정적·실천적 차원의 명확한 이전/이후                   |
| 논제         | 한 문장, 반통념적, 방어 가능                            |
| 저자 각도    | 적어도 하나의 설득력 있는 신뢰성 원천                   |
| 관건         | 긴급성이 명확 — 독자가 행동하지 않는 비용을 느낌        |
| 핵심 개념    | 3-7개 우선순위화, 각각 명확히 논제를 지지               |
| 적           | 명명되고 구체적                                         |
| 약속         | 독자가 매력적으로 느낄 한 문장                          |

## 출력물

모든 북 콘셉트 문서에 `assets/templates/book-concept-template.md` 템플릿을 사용합니다.

문서는 버전 관리(v1, v2 등)되며 포함하는 것:

- 모든 8가지 요소와 현재 개발 상태
- 진행 추적 세션 로그
- 향후 세션을 위한 미해결 질문
- 다운스트림 스킬을 위한 준비도 평가

## 구조적 프레임워크 참고

핵심 개념을 논의할 때, 잠재적 책 구조를 미리 보여주는 것이 종종 도움됩니다. 사용자와 공유할 프레임워크 옵션은 `references/nonfiction-frameworks.md`를 참조하세요. 이것은 방향 잡기 전용 — 구조적 결정은 `book-architect`의 영역입니다.

## 다운스트림 스킬로의 핸드오프

북 콘셉트 문서가 준비되면:

→ **idea-validator**: 논제와 핵심 개념을 기존 연구와 대조하여 스트레스 테스트
→ **market-research**: KDP 실행 가능성, 경쟁, 포지셔닝 평가
→ **book-architect**: 독자/변환/핵심 개념을 사용하여 구조 설계
