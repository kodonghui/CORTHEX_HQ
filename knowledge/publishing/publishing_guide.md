# 출판·기록 본부 가이드라인

> 이 문서는 출판·기록 본부의 모든 에이전트가 참고하는 공유 지식입니다.

---

## 우리 본부의 미션

**"기록되지 않은 것은 존재하지 않은 것이다."**

CORTHEX HQ가 만들어지는 과정, CEO의 의사결정, 시행착오, 성장 과정을
체계적으로 기록하고 매력적인 콘텐츠로 만들어 세상에 공유합니다.

---

## 기록 대상

### 1. 회사 빌딩 여정
- 왜 AI 에이전트 회사를 만들기로 했는가
- 어떤 기술적 선택을 했고, 왜 그렇게 결정했는가
- 조직 구조가 어떻게 진화했는가 (v0.1.0 → 현재)
- 실패한 시도와 그로부터 배운 교훈

### 2. 기술 아키텍처 변화
- 멀티 에이전트 시스템의 설계 철학
- 본부장 제거 결정 (v0.2.0)의 배경과 효과
- 딥워크(자율 멀티스텝) 도입 (v0.6.0)의 이유
- 비용 최적화 전략 (모델 계층 분리)

### 3. CEO의 비전과 철학
- "AI가 진짜 회사처럼 일할 수 있을까?"라는 질문
- 사용자 경험에 대한 생각
- 미래 방향성과 로드맵

## 콘텐츠 유형별 가이드

### 빌딩 로그 (Building Log)
- **형식:** 날짜 기반 일지
- **톤:** 솔직하고 날것 그대로. "이거 왜 안 되지?" 같은 실시간 고민도 OK
- **길이:** 500~1500자
- **필수 포함:** 날짜, 한 일, 고민한 것, 배운 것

### 기술 블로그
- **형식:** 문제 → 해결 과정 → 결과 → 교훈
- **톤:** 전문적이지만 초보자도 이해 가능하게
- **길이:** 1500~3000자
- **필수 포함:** 코드 예시 또는 다이어그램, 구체적 수치

### 회고/에세이
- **형식:** 자유 형식
- **톤:** 개인적이고 성찰적
- **길이:** 800~2000자
- **필수 포함:** "그때 왜 그렇게 생각했는지"에 대한 솔직한 고백

---

## 작성 원칙

1. **구체적으로:** "많이 개선됐다" ❌ → "LLM 호출이 4회에서 3회로 줄어 비용 25% 절감" ✅
2. **왜를 빠뜨리지 않기:** 무엇을 했는지보다 왜 그렇게 했는지가 더 중요
3. **실패도 기록:** 실패한 시도가 성공한 시도보다 더 좋은 콘텐츠
4. **독자를 생각하기:** "AI로 회사를 만드는 과정"에 관심 있는 사람이 읽는다고 가정
5. **시간 순서 유지:** 나중에 타임라인으로 정리할 수 있도록
