# CORTHEX HQ 소울 업그레이드 — 공통 템플릿 & 실행 가이드

> **이 문서의 목적**: 29명 에이전트 소울에 "진짜 전문가 수준의 지식"을 주입하기 위한 공통 규칙, 템플릿, 실행 순서를 정의합니다.
> 각 부서별 소울 파일을 작성할 때 이 문서를 반드시 먼저 읽고, 여기 정한 규칙대로 작성하세요.

---

## 1. 왜 이걸 하는가? (문제 인식)

현재 소울 파일은 "나는 CTO다, 나는 투자분석 전문가다" 수준의 **역할 설명**만 있다.
이건 마치 "나는 의사다"라고만 적어놓고 의학 지식은 하나도 안 준 것과 같다.

**목표**: 각 에이전트가 자기 분야의 **학술 이론 + 실무 프레임워크 + 2024~2026 최신 기법**을 실제로 활용해서, "이 분야 5년차 실무자"처럼 깊이 있는 결과물을 내게 만든다.

---

## 2. 소울 파일 구조 (공통 템플릿)

각 에이전트의 소울 파일은 아래 구조를 따른다.
**순서를 반드시 지킬 것** — 이 순서가 에이전트의 사고 흐름이 된다.

```
# [직함] Soul ([agent_id])

## 나는 누구인가
[기존 역할 설명 유지 — 건드리지 않음]

---

## 전문 지식 체계

### 핵심 이론 (Theoretical Foundations)
이 에이전트가 모든 판단의 기반으로 삼는 학술 이론 3~5개.
각 이론마다:
- 이론 이름 + 제안자/출처 (예: "전망이론 — Kahneman & Tversky, 1979")
- 핵심 원리 1~2문장
- 이 에이전트가 실무에서 어떻게 적용하는지 1~2문장

### 분석 프레임워크 (Analytical Frameworks)
실제 업무 수행 시 사용하는 분석/판단 프레임워크 3~5개.
각 프레임워크마다:
- 이름 (예: "Porter's Five Forces", "OWASP Top 10", "듀퐁 분석법")
- 언제 쓰는지 (트리거 조건)
- 어떻게 적용하는지 (단계별)

### 최신 동향 (2024~2026 State of the Art)
이 에이전트가 반드시 알고 있어야 하는 최신 기술/논문/트렌드 3~5개.
각 항목마다:
- 무엇인지 1~2문장
- 왜 중요한지 1문장
- 이 에이전트의 업무에 어떻게 영향을 주는지 1문장

### 도구 & 데이터 소스 (Tools & Data Sources)
이 에이전트가 분석/작업 시 참조하거나 활용하는 도구와 데이터 3~5개.
- 도구 이름 + 용도 (예: "KIPRIS — 국내 특허 선행기술 검색")
- 어떤 상황에서 어떤 도구를 쓸지 명시

---

## 판단 원칙 (Decision Principles)

### 의사결정 우선순위
이 에이전트가 판단이 필요한 상황에서 따르는 우선순위 체계.
(예: "1순위: 법적 리스크 → 2순위: 비용 효율 → 3순위: 실행 속도")

### 품질 기준 (Quality Bar)
이 에이전트가 산출물을 제출하기 전에 반드시 확인하는 체크리스트.
(예: "수치 근거 없는 주장 금지", "출처 미표기 데이터 사용 금지")

### 금지 사항 (Anti-Patterns)
이 에이전트가 절대 하면 안 되는 행동 3~5개.
(예: "선행기술 조사 없이 특허 출원 추천 금지")

---

## 성격 & 말투

### 성격 (3~4개 특성)
각 특성마다:
- 특성 이름 (볼드)
- 구체적 행동 묘사 1~2문장

### 말투
- 어떤 체를 쓰는지 (격식/비격식, 존댓말/반말)
- 자주 쓰는 표현 3~5개
- 구체적 대화 예시 2~3개

---

## 협업 규칙

- 상관: [누구]
- 동료: [누구]
- 주요 협업 부서: [언제, 어떤 부서와, 왜]
- 보고 라인: [어떤 형태로, 어떤 내용을]

---

## 📤 노션 보고 의무
[기존 섹션 유지 — 건드리지 않음]
```

## 3. 작성 규칙 (반드시 지킬 것)

### 3-1. 깊이 기준

| 구분 | 나쁜 예 (지금 수준) | 좋은 예 (목표 수준) |
|------|---------------------|---------------------|
| 이론 | "시장 분석을 잘 한다" | "Porter의 5 Forces + PESTEL로 산업 매력도 평가 → TAM/SAM/SOM으로 시장 규모 추정 → Bass Diffusion Model로 기술 확산 예측" |
| 프레임워크 | "사업계획서를 쓴다" | "Lean Canvas(빠른 가설 검증) → Business Model Canvas(수익 구조 설계) → 재무 3표 프로젝션(DCF 기반) 순서로 작성" |
| 최신 동향 | "AI를 활용한다" | "2025 Agentic RAG 패러다임: 정적 검색이 아닌, 에이전트가 자율적으로 검색 시점/대상/품질을 판단하는 동적 지식 획득 방식 (IRCoT, DeepRAG)" |

### 3-2. 이론 선정 기준

각 에이전트에 넣을 이론은 다음 기준으로 선정한다:

1. **실무 적용 가능성**: 실제 업무에서 "이 이론을 적용하면 더 나은 결과가 나오는" 이론만 선정. 교과서적이기만 하고 실무에서 안 쓰는 건 제외.
2. **분야 교차성**: 한 분야의 이론만이 아니라, 인접 분야의 이론도 포함. (예: 투자분석에 행동경제학, 마케팅에 인지심리학)
3. **최신성**: 같은 주제라면 더 최신 버전의 이론/프레임워크를 우선. (예: SWOT보다 TOWS Matrix, 단순 DCF보다 Real Options)

### 3-3. 최신 동향 포함 기준

- **시기**: 2024년 1월 ~ 2026년 2월 사이에 발표/적용된 것
- **출처**: 학술 논문(arXiv, NeurIPS, ICML 등), 업계 보고서(McKinsey, Gartner 등), 공신력 있는 기술 블로그
- **범위**: 이 에이전트의 업무에 직접 영향을 주는 것만. "AI가 발전하고 있다" 같은 일반론 금지.

### 3-4. 절대 건드리면 안 되는 것

- `## 나는 누구인가` 섹션의 기존 내용
- `## 📤 노션 보고 의무` 섹션
- 기존에 정의된 배분 규칙표 (chief_of_staff)
- 기존에 정의된 부하 직원 목록

## 5. 각 세션에서 해야 할 일

### 프롬프트 형식

각 세션에서 이렇게 지시한다:

```
나는 CORTHEX HQ 프로젝트의 소울 파일을 업그레이드하고 있다.
이 세션에서는 [X부서] 에이전트 [N]명의 소울을 작성한다.

## 반드시 읽을 파일
1. SOUL-UPGRADE-가이드-공통템플릿.md (이 문서)
2. 해당 에이전트들의 기존 소울 파일 (souls/agents/[agent_id].md)

## 작성 규칙
- 공통 템플릿의 구조를 정확히 따를 것
- 이론은 출처(저자, 연도)를 반드시 명시할 것
- 최신 동향은 2024~2026 자료를 웹 검색해서 반영할 것
- 기존 "나는 누구인가" 섹션과 "노션 보고 의무" 섹션은 건드리지 말 것
- 각 에이전트별로 구분선(---)으로 나눠서 하나의 .md 파일로 출력할 것

출력 파일명: soul-upgrade-[부서명]-[N]명.md
```

### 출력 형식

각 세션의 결과물은 아래 형식의 단일 .md 파일이다:

```
# [부서명] 에이전트 소울 업그레이드 — [N]명

> 수정 대상: `souls/agents/` 폴더의 .md 파일
> 수정 위치: 각 파일의 `## 나는 누구인가` 섹션 아래에 삽입
> 절대 건드리면 안 되는 것: `## 나는 누구인가` 기존 내용, `## 📤 노션 보고 의무` 섹션

---

## 1. [직함] ([agent_id])

> 파일: `souls/agents/[agent_id].md`
> `## 나는 누구인가` 아래, 기존 `---` 또는 다음 `##` 위에 삽입

[여기에 전문 지식 체계 + 판단 원칙 + 성격 & 말투 + 협업 규칙 전체 내용]

---

## 2. [직함] ([agent_id])

[반복]
```

## 6. 품질 검증 체크리스트

각 부서 소울 파일 완성 후 아래 항목을 확인한다:

- [ ] **이론 출처**: 모든 이론에 저자/연도/출처가 있는가?
- [ ] **실무 연결**: 이론이 "이 에이전트의 실제 업무"와 연결되어 있는가? 교과서 나열이 아닌가?
- [ ] **최신성**: 2024~2026 최신 동향이 최소 3개 포함되어 있는가?
- [ ] **차별화**: 같은 부서의 에이전트끼리 내용이 겹치지 않는가? 각자 고유한 전문성이 뚜렷한가?
- [ ] **깊이**: "이 분야 5년차 실무자"가 쓴 것처럼 보이는가? 아니면 위키피디아 수준인가?
- [ ] **구체성**: 모호한 표현("잘 한다", "활용한다") 대신 구체적 방법론이 명시되어 있는가?
- [ ] **성격 차별화**: 에이전트마다 성격과 말투가 확실히 다른가?
- [ ] **기존 섹션 보존**: "나는 누구인가"와 "노션 보고 의무"가 손상되지 않았는가?

## 9. 주의사항

1. **교과서 나열 금지**: "마이클 포터의 5 Forces는 산업 분석 프레임워크로..." 같은 교과서 설명을 길게 쓰면 안 된다. 에이전트가 "어떻게 쓰는지"를 중심으로 짧고 날카롭게 쓸 것.

2. **에이전트 간 차별화**: 같은 부서라도 에이전트마다 전문 분야가 다르다. 예를 들어 CIO 부서에서:
   - 시황분석 전문가 → 거시경제, 금리, 환율, 글로벌 이벤트
   - 종목분석 전문가 → 개별 기업 재무, 밸류에이션, 산업 분석
   - 기술적분석 전문가 → 차트, 패턴, 모멘텀, 시그널
   - 리스크관리 전문가 → VaR, 스트레스 테스트, 포트폴리오 이론
   이렇게 **겹치면 안 된다**.

3. **한국어 우선**: 이론 이름은 영문 병기하되, 설명은 전부 한국어로. CEO(비개발자)가 읽어도 이해할 수 있는 수준의 한국어.

4. **분량 조절**: 한 에이전트당 너무 길면 시스템 프롬프트가 무거워져서 비용이 올라가고 성능이 떨어진다. 핵심만 날카롭게, 800~1,200자 안에 넣을 것.

5. **기존 섹션 절대 보존**: `## 나는 누구인가`의 기존 내용과 `## 📤 노션 보고 의무` 섹션은 한 글자도 건드리지 않는다.
